# 个人博客开发规划

## 1. 项目目标

开发一个纯前端个人博客网站，基于 Nuxt.js 构建，通过静态站点方式部署到腾讯云服务器。网站不依赖后端服务，内容以本地文件、静态资源和前端状态管理为主，后续可以逐步扩展联系信息、个人展示、知识库、Todo 规划等模块。

一句话定位：

> 这是一个高级、克制、文件驱动的个人知识展示站，而不是网页端笔记软件、任务管理软件或在线 CMS。

核心目标：

- 建设一个视觉优美、适合长期维护的个人博客首页。
- 建设一个高级、克制、有个人识别度的界面系统，避免模板感和廉价动效。
- 支持 Wiki 知识库，用于沉淀笔记、文档、学习资料。
- 支持 Todo 规划模块，用于展示计划、阶段目标和任务清单。
- 支持背景更换、站点设置、音乐播放等个性化功能。
- 增加一只全站常驻的小狗宠物助手，点击后弹出快捷列表，可进入 AI 问答和音乐播放。
- 兼容桌面端、平板和移动端访问。
- 保留专门存放 Wiki 笔记和 Todo 文件的目录，方便后续直接添加内容。
- 明确网站是纯前端展示层，不提供网页端写笔记、编辑 Wiki、修改 Todo、发布文章等内容写入能力。
- 内容生产流程固定为本地 Markdown/JSON 编辑、Git 管理、重新构建和静态部署；博客和 Wiki 统一使用 Markdown。
- 采用纯网页形式，不引入后端数据库、登录系统或服务器接口。

第一版必须坚持的产品边界：

- 页面可以像知识工作台一样高级，但不能让用户误以为可以在浏览器里写入正式内容。
- 博客和 Wiki 的核心内容只来自仓库中的 `content/` Markdown 文件；Todo 来自 `content/todo/` Markdown 或 JSON 文件，不来自浏览器输入框、`localStorage` 或临时状态。
- 网页端允许做搜索、筛选、排序、目录跳转、阅读进度、折叠展开、主题背景偏好、音乐播放和 AI 问答入口。
- 网页端不允许做正式内容的新增、编辑、删除、保存、发布、勾选完成、拖拽改状态。
- 如果某个交互会产生“我已经在网页上保存了笔记或 Todo”的错觉，第一版不要做。
- 所有可持久化的前端状态都只能是偏好状态，例如主题、背景、播放器开合、小狗显示、面板位置，不能是正式文章、Wiki 笔记或 Todo 内容。

## 2. 技术选型

推荐技术栈：

- 框架：Nuxt.js 3
- 语言：TypeScript
- 样式：Tailwind CSS 或 UnoCSS
- 内容管理：Nuxt Content
- 状态管理：Pinia
- 图标：lucide-vue-next
- 音频播放：明确采用 APlayer + MetingJS，使用 MetingJS 解析 QQ 音乐歌单，APlayer 负责播放界面和控制能力
- 动画：CSS Transition、Vue Transition，必要时使用 Motion One
- AI 问答：优先设计为可插拔模块，第一版避免在前端硬编码任何私密 API Key
- 部署方式：Nuxt 静态生成后上传到腾讯云服务器

推荐渲染方式：

- 使用 `nuxi generate` 生成静态文件。
- 服务器只托管静态资源，不运行 Node 服务。
- Nginx 指向生成后的静态目录，例如 `.output/public`。

### 外部博客与静态站调研结论

本项目学习的是成熟博客的设计原则和内容工作流，不照搬任何站点的品牌、素材和代码。

参考对象与可借鉴点：

- [Josh W. Comeau](https://www.joshwcomeau.com/)：文章列表、分类、热门内容和互动教学组件很完整；他在博客构建文章中提到使用 MDX、frontmatter、在 VS Code 中编辑并提交，适合作为“内容在文件中，网站负责渲染”的参考。
- [Lee Robinson](https://leerob.com/)：首页极简，先讲清楚身份和代表内容，再进入写作链接；可借鉴“短介绍 + 精选文章”的信息组织。
- [Rauno Freiberg](https://rauno.me/)：极少元素也能形成高级感，关键在速度、美感、一致性、细节和长期主义；可作为界面克制程度的参考。
- [Steph Ango](https://stephango.com/)：文章、主题、归档非常清楚；他的 “File over app” 与 “Style is consistent constraint” 思路适合作为本项目的内容哲学：文件优先、风格来自一致约束。
- [Everything I Know by Owen](https://wiki.owenyoung.com/)：个人知识库以最近更新、标签、Wiki、图谱等方式组织内容；可借鉴 Wiki 的索引、标签和最近更新，而不是做网页端编辑器。
- [Anthony Fu](https://antfu.me/)：个人主页把身份、项目、博客、演讲和链接组织在一个轻量体系中；适合参考开源开发者个人站的信息密度。
- [Vercel Blog Templates](https://vercel.com/templates/blog)：大量模板采用 Markdown、静态生成、无数据库或轻 CMS；说明静态博客的核心价值是发布和阅读，而不是浏览器内编辑。
- [AS Notes 静态发布流程](https://blog.asnotes.io/as-notes-a-static-site-generator-in-your-markdown-knowledgebase)：强调在 VS Code 写 Markdown、通过 Git 推送触发静态发布；适合本项目的维护流程。
- [Maggie Appleton 的 Digital Garden 文章](https://maggieappleton.com/garden-history)：数字花园强调长期生长、链接和持续修订；适合 Wiki 模块的内容组织气质。

转化为本站原则：

- 首页可以有强记忆点，但内容页必须像高质量阅读产品。
- 高级感来自约束、留白、排版、细节和一致性，不来自堆卡片、堆阴影、堆特效。
- 博客和 Wiki 都是从 Markdown 生成的只读页面；Todo 从 Markdown 或 JSON 生成；页面只提供筛选、搜索、导航、播放、问答和设置，不提供内容写入。
- Todo 页面是规划展示，不是可持久化任务应用；不要做点击完成、拖拽改状态、网页新增任务等会误导用户的交互。
- Wiki 页面是知识库阅读和检索，不是在线 Notion、Obsidian 或 CMS。
- 可以使用 `localStorage` 保存主题、背景、播放器、小狗隐藏等偏好，但不能把它当成内容存储方案。

### 非目标与禁止项

为了避免纯前端项目被做成无法真正保存的“假编辑器”，第一版明确不做以下内容：

- 不做网页端写笔记、网页端新增 Wiki、网页端新建博客文章。
- 不做网页端 Todo 勾选完成、拖拽排序、修改状态并声称保存。
- 不做发布按钮、保存按钮、提交文章按钮、同步到云端按钮。
- 不做登录注册、后台管理、数据库、服务端内容接口。
- 不做没有后端保护的联系表单、评论系统和私密 AI Key 直连。
- 不把 `localStorage` 包装成正式内容存储；它只用于 UI 偏好和临时会话。

可以做的替代体验：

- 用“最近更新”“标签”“目录树”“阅读进度”“相关内容”提升知识库体验。
- 用“路线图”“状态摘要”“阶段进度”“时间线”展示 Todo，而不是做可编辑看板。
- 用“编辑此内容的维护说明”引导自己回到本地 `content/` 文件修改，而不是在网页上写入。
- AI 第一版可以推荐本地相关文章和 Wiki，不把 AI 面板设计成内容保存入口。

│   └── assets/

目录说明：

- `docs/`：项目规划、部署说明、设计规范、维护文档。
- `docs/blog-development-todolist.md`：面向开发执行的详细 Todo 清单，按照阶段、文件和验收点拆分。
- `content/wiki/`：Wiki 知识库 Markdown 笔记目录，需要保留；只能通过本地编辑文件维护，网页端只读展示。
- `content/todo/`：Todo 规划 Markdown 或 JSON 文件目录，需要保留；网页端只读展示，不做任务状态写入。
- `content/blog/`：博客文章目录；通过本地 Markdown 编辑、提交和重新部署发布，不使用 MDX。
- `public/backgrounds/`：背景图片资源根目录。
- `public/backgrounds/home/day/`：博客主页白天模式专用背景，后续把浅色、清爽、明亮的首页背景放这里。
- `public/backgrounds/home/night/`：博客主页黑夜模式专用背景，后续把暗色、低亮度、适合夜间阅读的首页背景放这里。
- `public/backgrounds/shared/`：可被多个页面复用的通用背景或纹理，不和主页专用背景混放。
- `public/pet/`：小狗宠物助手的图片、序列帧、Lottie 或轻量动画资源。
- `public/music/`：音乐文件资源。
- `public/images/`：头像、插图、封面等公共图片资源。

组件建议：

- `app/components/home/MimoInspiredHero.vue`：首页首屏动效，学习 Xiaomi MiMo 官网的重复字母背景、中央圆形遮罩和大标题穿插效果。
- `app/components/home/HomeLetterField.vue`：首屏背景字母矩阵，可根据站点名称或关键词生成重复字符。
- `app/components/pet/PetDogAssistant.vue`：全站常驻小狗入口。
- `app/components/pet/PetActionMenu.vue`：点击小狗后出现的快捷列表。
- `app/components/ai/AiChatPanel.vue`：AI 问答抽屉或弹层。
- `app/components/music/MusicPlayer.vue`：完整音乐播放器。
- `app/components/music/MiniMusicPlayer.vue`：移动端或收起状态播放器。
- `app/composables/usePetAssistant.ts`：管理小狗状态、菜单开合、动画状态。
- `app/composables/useMusicPlayer.ts`：管理播放列表、播放状态、音量。
- `app/composables/useSiteSettings.ts`：管理背景、主题、动效、字体偏好。
- `app/config/backgrounds.ts`：维护主页白天/黑夜背景列表、默认背景、缩略图、遮罩强度和适用主题。
- `app/config/ai.ts`：只维护模型名称、代理地址等非敏感配置，私密 Key 只能来自服务端环境变量或用户本地输入。
- `app/config/music.ts`：维护 APlayer/MetingJS 配置、QQ 音乐歌单 ID、播放器模式和外部资源开关。

## 4. 页面规划

### 首页

首页作为博客的第一入口，应直接展示个人博客的核心气质，而不是单纯的导航页。

建议包含：

- 个人名称、博客名称或站点标题。
- 当前精选文章、最近 Wiki 更新、近期 Todo 状态。
- 简洁导航入口：博客、Wiki、Todo、关于、联系。
- 背景图或动态视觉区域。
- 右下角或固定区域放置音乐播放器入口。
- 设置按钮用于切换背景、主题和显示偏好。
- 小狗宠物助手常驻在页面右下角或左下角，作为 AI 问答和音乐播放的趣味入口。

主页背景资源规划：

- 博客主页背景单独放在 `public/backgrounds/home/`，不要和文章封面、头像、小狗资源混在一起。
- 白天模式背景放在 `public/backgrounds/home/day/`，适合浅色主题，画面亮度高但不能影响标题可读性。
- 黑夜模式背景放在 `public/backgrounds/home/night/`，适合深色主题，避免高亮大面积刺眼区域。
- 背景可以先留空目录，后续你把图片加进去后，在 `app/config/backgrounds.ts` 中登记文件路径、名称、适用主题和遮罩强度。
- 首页允许使用更有氛围的背景，但 Blog、Wiki、Todo 内容页默认使用更克制的纯色或轻纹理背景，保证阅读。

MiMo 风格首屏学习方向：

- 首页第一屏学习 Xiaomi MiMo 官网的视觉语言：浅色背景、重复字母矩阵、中央大黑圆、超大标题横向穿过圆形区域。
- 只学习动效和构图，不直接使用 Xiaomi MiMo 的 Logo、品牌文案和素材；标题替换成个人博客身份，例如 `你好，我是 Cherish` 或博客名称。
- 字母矩阵可以从 `NOTEBOOK`、`BLOG`、站点名缩写或个人英文名生成，使用低透明度、大字距、固定行高铺满首屏。
- 白天模式下字母矩阵使用低透明度深色文字；黑夜模式下使用低透明度浅色文字，不能抢过主标题。
- 中央黑色圆形作为视觉焦点，可使用 CSS `border-radius: 50%`、`transform`、`clip-path` 或 `mask-image` 实现；桌面端可跟随鼠标做轻微位移和缩放。
- 大标题分两层处理：普通黑色文字层和圆形内的白色反相文字层，优先使用 `mix-blend-mode: difference`、遮罩裁切或双文本叠层实现类似穿插效果。
- 首屏高度建议使用 `min-height: min(760px, 100svh)`，底部露出下一屏内容提示，避免首页只停留在纯视觉封面。
- 导航保持极简，左侧放站点 Logo 或名称，右侧放 `Blog`、`Wiki`、`Todo`、`关于`、语言或主题按钮。
- 滚动时黑圆可以略微放大、标题淡出，过渡到精选文章、Wiki 更新和 Todo 状态区域。
- 移动端标题改为两行或三行，圆形尺寸使用 `clamp(220px, 72vw, 520px)`，避免文字溢出。
- 尊重 `prefers-reduced-motion`：关闭鼠标跟随、视差和循环动画，只保留静态构图。
- 字母背景设置 `aria-hidden="true"`，真实页面标题仍使用语义化 `h1`，保证可访问性和 SEO。
- 白天/黑夜模式都必须单独验收首页首屏，确保背景、字母矩阵、圆形和标题对比度足够。

### 全站小狗宠物助手

这是网站的标志性交互组件，建议作为全站固定入口，而不是只放在首页。

核心定位：

- 小狗是博客的轻量助手，负责承载 AI 问答、音乐播放和后续快捷功能。
- 默认以小尺寸悬浮在页面角落，不干扰阅读。
- 鼠标悬停或轻触时，小狗有抬头、摇尾巴、发光描边或轻微弹跳动画。
- 点击小狗后出现一个小型快捷列表，提供 `AI 问答`、`音乐播放`、`站点设置` 等入口。
- 当音乐播放中，小狗可以显示节奏感状态，例如耳机、音符、轻微律动。
- 当 AI 正在回复时，小狗可以进入思考状态，例如眼睛闪烁、头顶小气泡、加载动画。

交互要求：

- 桌面端：小狗悬浮在右下角，点击后菜单向上展开。
- 移动端：小狗尺寸缩小，菜单改为底部弹层或紧凑浮层。
- 阅读文章和 Wiki 时，小狗不能遮挡正文、目录、代码块和滚动条。
- 支持关闭或隐藏，隐藏状态保存在 `localStorage`。
- 支持减少动效模式，尊重系统 `prefers-reduced-motion`。

快捷列表建议：

```text
小狗助手
├── AI 问答
├── 音乐播放
├── 换个背景
└── 设置
```

视觉方向：

- 小狗形象要有辨识度，可以使用 2D 插画、像素风、轻 3D 风或 Lottie 动画。
- 菜单使用半透明玻璃质感、细边框、柔和阴影和短动效，突出酷炫但保持克制。
- 动画重点放在小狗本身和菜单展开，不把正文区域做得过度花哨。
- 夜间主题下小狗可以有微弱描边光效，白天主题下使用更干净的阴影和高光。
- 小狗菜单只作为入口，不承载“写笔记”“新增 Todo”“发布文章”等写入动作。

## 5. 功能规划

### 小狗宠物助手

功能：

- 全站悬浮展示一只小狗。
- 点击小狗打开快捷列表。
- 快捷列表中可以选择 `AI 问答` 和 `音乐播放`。
- 可扩展 `换背景`、`设置`、`回到顶部`、`随机文章` 等动作。
- 小狗根据状态变化：空闲、悬停、菜单打开、音乐播放、AI 思考、隐藏。

状态设计：

```text
idle       空闲巡航或轻微呼吸动画
hover      鼠标悬停，抬头或摇尾巴
open       快捷列表展开
music      音乐播放中，有轻微律动
thinking   AI 回复中，显示思考动画
hidden     用户隐藏小狗
```

实现建议：

- 第一版可以用静态 PNG/WebP 加 CSS 动画实现，降低开发复杂度。
- 如果追求更酷炫的效果，后续可以替换为 Lottie、Spine、Rive 或 Canvas 动画。
- 小狗组件放在全局 layout 中，保证各页面都能访问。
- 小狗菜单只负责打开功能面板，不在小菜单里塞入复杂内容。
- 动画时间控制在 120ms 到 280ms，避免影响操作效率。
- 需要提供关闭按钮或设置项，避免用户阅读时被打扰。

### AI 问答

目标：

- 通过小狗助手进入 AI 问答面板。
- 支持用户询问 Wiki 笔记、博客内容、Todo 规划或普通问题。
- 第一版先完成交互壳、聊天界面、输入框、消息列表和加载状态。
- 后续根据安全方案接入真实 AI 模型。

当前模型补充：

- 模型服务地址：`http://3stooges.chat:4036`（原始记录里少了一个 `/`，实现前需要确认实际地址是否为 `http://`）。
- 模型名称：`minimax-m2.7-awq`。
- API Key：只允许放在本地 `.env.local`、服务器环境变量、云函数环境变量或用户浏览器本地输入中，不能提交到仓库，也不能写进前端构建产物。
- 前端统一请求自己的安全代理，例如 `/api/ai/chat` 或云函数地址；代理再读取环境变量调用模型服务。
- 如果坚持纯静态部署，真实模型调用必须走外部安全代理或用户自填 Key 模式，不能由静态前端直接携带私密 Key 请求模型。

推荐本地私有配置示例：

```bash
AI_BASE_URL=http://3stooges.chat:4036
AI_MODEL=minimax-m2.7-awq
AI_API_KEY=写入本地私密 Key，禁止提交
```

纯前端安全边界：

- 不能把私密 API Key 写进前端代码或公开配置文件。
- 静态网站里的任何环境变量和请求参数都可能被访问者看到。
- 如果要接入真实模型，优先考虑安全代理、Serverless Function 或用户自填 Key。

推荐接入路线：

1. 本地知识库问答：先基于 `content/wiki/` 生成索引，提供关键词搜索、相关文章推荐和摘要展示，不调用外部 AI。
2. 用户自填 Key：只适合个人自用或小范围使用，Key 存在浏览器本地，页面明确提示风险。
3. 安全代理模式：后续使用腾讯云函数、Edge Function 或轻量服务转发 AI 请求，前端不暴露私密 Key。

AI 面板建议：

- 以右侧抽屉或底部弹层打开。
- 展示当前问题、回答、引用的 Wiki 或博客文档。
- 支持清空会话。
- 支持把当前文章标题或 Wiki 页面作为上下文。
- 移动端使用全屏抽屉，输入框固定在底部。
- AI 回答可以提供“相关内容推荐”和“复制回答”，但第一版不提供“保存到 Wiki”“生成新文章”“写入 Todo”等持久化按钮。

### 小狗快捷音乐入口

小狗菜单中的 `音乐播放` 应与完整音乐播放器联动。

交互：

- 点击 `音乐播放` 后打开迷你播放器或完整播放面板。
- 如果当前没有播放，展示播放列表和主播放按钮。
- 如果正在播放，小狗进入 `music` 状态。
- 关闭播放器时不一定停止音乐，播放状态由播放器自己管理。
- 播放器可以收起为迷你条，小狗仍显示音乐状态。

建议体验：

- 播放按钮、上一首、下一首、音量、列表入口使用图标按钮。
- 当前歌曲名称在空间不足时使用省略号，不挤压布局。
- 移动端播放器贴底显示，并避开系统安全区域。
- 浏览器禁止自动播放时，提示用户点击小狗或播放按钮开始。

### 音乐模块实现方案

音乐模块明确采用 `APlayer + MetingJS` 实现：

- 在 Nuxt 全局入口中挂载一个独立音乐播放器组件，让播放器跨页面存在，切换路由时不中断。
- 通过 `nuxt.config.ts` 的 `app.head.script` 和 `app.head.link` 全局加载 APlayer、MetingJS 和播放器样式。
- 在 Vue 编译配置中把 `meting-js` 声明为自定义元素，避免 Vue 把它当成未知组件处理。
- 播放器组件内部用 `ready` 状态延迟渲染 `meting-js`，减少客户端挂载时序问题。
- 用 `localStorage` 保存播放器展开或收起状态。
- 用固定定位把播放器放在页面角落，通过 CSS transform 做展开、收起和移动端适配。
- 额外写一份深色模式样式覆盖 APlayer 默认皮肤，让播放器和站点主题一致。
- MetingJS 的 `server` 使用 QQ 音乐对应的 `tencent`，`type` 使用 `playlist`，`id` 等你后续提供 QQ 音乐歌单后再填入配置。
- QQ 音乐歌单优先使用公开歌单；如果歌单不可公开访问或歌曲因版权无法播放，播放器需要降级展示提示或切换到本地备用歌单。

整体交互不照搬普通音乐播放器的入口方式。你的站点应保留小狗宠物助手作为核心入口：

- 不使用单独的普通音乐展开按钮，音乐入口放进小狗快捷列表。
- 小狗点击后展开酷炫快捷菜单，选择 `音乐播放` 后再打开音乐面板。
- 音乐播放中，小狗进入律动状态，显示耳机、音符、光圈或频谱动效。
- 音乐面板可以使用 APlayer 的成熟播放能力，但外层视觉重新包装成你的站点风格。
- APlayer/MetingJS 的默认皮肤需要重写为暗色玻璃、霓虹描边、柔和光效和移动端底部抽屉。
- 播放器状态要和 `useMusicPlayer`、`usePetAssistant` 打通，避免小狗和播放器各管各的状态。

推荐实现路线：

1. 先做 `PetDogAssistant.vue` 和 `PetActionMenu.vue`，把 `音乐播放` 作为菜单项。
2. 再做 `MusicPlayer.vue`，内部封装 APlayer/MetingJS，不再手写完整音频播放逻辑。
3. 把外部脚本、样式、音乐平台、QQ 音乐歌单 ID、Meting API 地址和备用歌单都放进 `app/config/music.ts`。
4. 所有私密信息都不能进入公开仓库；歌单 ID、公开 CDN 地址可以配置，私密模型 Key、代理鉴权信息不能配置在前端。
5. 最后做深色模式、移动端样式、小狗播放状态和播放器收起状态持久化。

### 背景更换

目标：

- 支持多张背景图切换。
- 支持白天模式、黑夜模式或沉浸式背景。
- 用户选择保存在浏览器 `localStorage`。

建议设计：

- 设置面板中提供背景缩略图。
- 支持默认背景。
- 背景配置放在 `app/config/backgrounds.ts`，按 `home.day`、`home.night`、`shared` 分组。
- 首页背景资源从 `public/backgrounds/home/day/` 和 `public/backgrounds/home/night/` 读取。
- 用户选择“跟随主题”时，白天模式自动使用 day 背景，黑夜模式自动使用 night 背景。
- 用户手动指定背景时，需要记录背景 ID 和主题适用范围；如果切到不适合的主题，使用默认背景兜底。
- 移动端背景需要单独检查裁切效果。
- 背景上方内容使用可读性遮罩，但避免过重的黑色蒙层。
- 可以从小狗快捷列表中直接触发 `换个背景`，随机切换一张背景。

白天与黑夜模式要求：

- 主题模式包含 `system`、`light`、`dark`，其中 `light` 对应白天模式，`dark` 对应黑夜模式。
- `system` 模式跟随系统 `prefers-color-scheme`，并自动切换对应背景组。
- 白天模式要保持清爽、明亮、文字对比清晰；黑夜模式要避免纯黑铺底和过亮背景。
- 背景图必须配套遮罩 token，例如 `overlayLight`、`overlayDark`，避免同一张图在不同主题下不可读。
- 背景切换只属于视觉偏好，不改变博客、Wiki、Todo 的正式内容。

### 音乐播放

功能：

- 播放、暂停。
- 上一首、下一首。
- 音量控制。
- 播放列表。
- 当前歌曲名称。
- 迷你播放器。
- 歌词展示，可选。
- 播放器展开、收起状态持久化。
- 与小狗助手共享播放中、暂停、加载、错误状态。
- 解析并播放你的 QQ 音乐歌单；歌单 ID 等到实际开发时再补充。
- 支持 QQ 音乐不可用时的本地备用曲目或空状态提示。

实现注意：

- 浏览器通常禁止未交互自动播放，因此音乐应在用户点击后开始。
- 本地备用音乐文件放在 `public/music/`。
- 移动端播放器保持小巧，避免遮挡正文。
- 与小狗助手共享播放状态，播放中小狗进入音乐状态。
- APlayer/MetingJS 相关参数优先通过可配置项管理，不把第三方实现细节散落在组件里。
- 如果使用外部音乐 API，需要考虑可用性、跨域、版权和服务稳定性；本地音乐文件方案更可控。
- 公开仓库中只保留公开播放列表或示例配置，私密令牌、个人模型信息和代理密钥一律不提交。

推荐配置结构：

```ts
export const musicConfig = {
  provider: 'meting',
  server: 'tencent',
  type: 'playlist',
  playlistId: '待补充',
  api: '默认或待配置',
  autoplay: false,
  fixed: false,
  mini: false,
  fallbackTracks: []
}
```

Nuxt 接入要点：

- `nuxt.config.ts` 中全局加载 APlayer CSS、APlayer 脚本和 MetingJS 脚本。
- `vue.compilerOptions.isCustomElement` 中声明 `meting-js` 为自定义元素。
- `MusicPlayer.vue` 通过 `<ClientOnly>` 或客户端挂载状态渲染，避免静态生成和服务端渲染阶段访问浏览器对象。
- 播放器外层由自己的面板、按钮、抽屉和小狗状态控制，内部的 `<meting-js>` 只负责歌单解析和 APlayer 初始化。

推荐组件职责：

- `MusicPlayer.vue`：负责真正的播放器渲染和播放面板。
- `MiniMusicPlayer.vue`：负责移动端或收起状态展示。
- `useMusicPlayer.ts`：负责播放状态、展开状态、当前曲目和本地存储。
- `PetDogAssistant.vue`：只展示小狗状态和入口，不直接承载复杂播放逻辑。
- `PetActionMenu.vue`：点击 `音乐播放` 时调用播放器打开方法。

## 6. 视觉设计方向

整体风格建议：

- 安静、清爽、有层次，不做复杂营销页风格。
- 背景可以有个人特色，但正文区域必须保证阅读体验。
- 首页可以更有氛围感，文章和 Wiki 页面应更克制。
- 组件圆角建议控制在 8px 左右，避免过度卡片化。
- 色彩使用 1 个主色、1 个强调色、多个中性色，避免全站单一色调。

建议页面气质：

- 首页：视觉化、个性化、带少量动态。
- 博客：清晰、舒适、适合长文阅读。
- Wiki：高密度、可扫描、重视导航效率。
- Todo：清楚展示状态和优先级。
- 小狗助手：酷炫、有记忆点，但始终服务于功能入口，不喧宾夺主。

### 高级界面设计规范

高级感的判断标准不是元素堆叠，而是信息层级清楚、材质克制、动效有理由、细节统一。界面要像一个完整的个人知识工作台，而不是多个组件拼在一起。

高级界面的核心目标：

- 第一眼要有个人站的识别度，不能像套模板。
- 第二眼要能快速阅读和定位内容，不能为了酷炫牺牲可用性。
- 长期使用时要安静、耐看、稳定，不依赖大面积炫光和复杂动效制造新鲜感。
- 所有功能看起来都应该是“正式产品的一部分”，包括空状态、错误状态、加载状态和移动端状态。

设计原则：

- 克制：首页可以有强视觉记忆点，内容页要把阅读效率放在第一位。
- 精准：每个阴影、边框、动效都要服务于层级，不做没有功能意义的装饰。
- 留白：页面使用明确的垂直节奏，避免元素贴边、拥挤和平均分布。
- 统一：导航、按钮、卡片、抽屉、弹窗、标签、输入框使用同一套尺寸、圆角、阴影和动效规则。
- 可读：正文、代码、目录、列表、任务状态始终比背景、宠物和播放器更重要。

布局与栅格：

- 页面主体使用 `max-width` 控制阅读宽度，首页内容区建议 `1120px` 到 `1280px`，长文正文建议 `720px` 到 `820px`。
- 桌面端采用 12 列或等价 CSS Grid；Wiki 和文章详情可以使用 `内容区 + 目录区` 的双栏布局。
- 移动端优先单列，不强行保留桌面端复杂结构。
- 首屏必须露出下一屏的一部分内容，提示页面可继续探索。
- 固定定位元素必须避开正文、目录、代码块、底部安全区和移动端浏览器工具栏。

字体与排版：

- 中文字体优先使用系统无衬线栈，保证跨平台清晰：`-apple-system`、`BlinkMacSystemFont`、`Segoe UI`、`PingFang SC`、`Microsoft YaHei`。
- 英文和数字可使用更利落的无衬线字体栈，标题里的英文可以稍微加粗，但不使用负字距。
- 正文字号桌面端建议 `16px` 到 `18px`，行高 `1.75` 左右；移动端正文不低于 `15px`。
- 标题层级最多保持 4 级视觉差异，避免每一级都很夸张。
- 首页 MiMo 风格大标题允许夸张，但内容卡片、面板和抽屉内的标题要收敛，保证专业感。

色彩与材质：

- 建立主题色、强调色、中性色、状态色四类 token，避免全站被单一颜色统治。
- 浅色主题以干净的近白背景、深色文字、低透明度线条为主。
- 深色主题避免纯黑大面积铺底，优先使用接近黑的中性色和低亮度边框。
- 玻璃质感只用于浮层、播放器、小狗菜单和设置面板，不用于所有内容卡片。
- 阴影必须轻，优先表现层级，不制造沉重的悬浮感。
- 霓虹、光圈、描边只作为音乐播放、小狗状态或首页焦点的局部反馈，不能全站泛滥。

组件视觉规范：

- 普通卡片圆角控制在 `8px` 左右，重复列表卡片保持简洁，不做厚重背景。
- 按钮按用途分为主操作、次操作、图标操作、危险操作，颜色和状态必须区分清楚。
- 工具按钮优先使用 lucide 图标，并提供 hover/focus/active/disabled 状态。
- 标签使用低饱和背景和细边框，不做大面积高亮。
- 表单输入框需要有聚焦态、错误态、禁用态和移动端触控高度。
- 抽屉、弹层、设置面板和 AI 面板要有一致的遮罩、关闭方式和键盘可访问性。

动效规范：

- 动效时间以 `120ms` 到 `280ms` 为主，首页首屏和小狗状态可以略长，但不能拖慢操作。
- 动效曲线使用自然的 ease-out 或 spring 风格，避免匀速和过度弹跳。
- 首页可使用鼠标跟随、视差、滚动过渡，但阅读页只保留必要反馈。
- 所有循环动效都要轻微，音乐播放状态可以律动，但不能影响正文阅读。
- 必须支持 `prefers-reduced-motion`，关闭视差、鼠标跟随、循环律动和大幅位移。

首页高级感要求：

- MiMo 风格首屏是首页记忆点，但个人博客身份必须更清楚，不能像仿站。
- 顶部导航要像产品官网一样干净，滚动前后状态要自然过渡。
- 首屏标题、中央圆形、背景字母三者必须有明确层级，不能互相抢。
- 下一屏直接进入精选内容和知识工作台，不做纯营销式空洞介绍。
- 首页的内容区可以使用分区、列表、时间线和状态摘要，避免大量浮动卡片堆叠。

博客、Wiki、Todo 高级感要求：

- 博客列表重视可读标题、摘要、日期、标签和阅读时间，避免封面图喧宾夺主。
- 文章详情重视阅读宽度、目录定位、代码块样式、引用样式和阅读进度。
- Wiki 页面重视目录树、当前路径、搜索入口和文档更新时间，让它像真实知识库。
- Todo 页面重视状态、优先级、时间和阶段，不做玩具化任务列表，也不做会误导用户以为能保存的勾选和拖拽。
- 所有内容模块都要有空状态、加载状态和错误状态，且文案短而清楚。

内容工作流高级感要求：

- 网站要像一个发布后的个人知识产品，而不是一个半成品后台。
- “写作、整理、改 Todo”发生在本地编辑器中；“阅读、搜索、导航、展示”发生在网页中。
- 所有写入型按钮都要谨慎，第一版不要出现 `新建文章`、`保存笔记`、`提交 Todo`、`发布` 等文案。
- 如果未来加入用户自填 Key 或临时草稿，也必须明确标注只保存在当前浏览器，不能混同于正式内容。
- 内容更新说明应写在关于页或维护文档中，让未来的自己知道如何新增 Markdown、更新 frontmatter、重新部署。

禁用交互样式：

- 不使用可编辑正文区域、富文本工具栏、Markdown 在线编辑器、保存浮条。
- Todo 不使用真实 checkbox 作为“完成任务”的交互控件；如需展示完成状态，使用只读状态标签或图标。
- Todo 不使用拖拽排序、看板列间拖动、行内状态编辑等会暗示可保存的交互。
- Wiki 不使用“编辑本页”“新增子页面”“上传附件”等 CMS 风格入口。
- Blog 不使用“发布文章”“保存草稿”“管理文章”等后台风格入口。

移动端质感要求：

- 移动端第一屏标题不能压住导航，也不能让小狗、播放器和设置按钮互相遮挡。
- 抽屉和底部弹层必须考虑安全区，底部按钮不贴边。
- 触控目标不小于 `44px`，图标按钮要有足够点击面积。
- 移动端导航不堆满顶部，优先使用紧凑菜单或底部动作入口。
- 长文阅读时隐藏或收起干扰元素，只保留必要导航和小狗入口。

设计验收方式：

- 每个页面至少检查桌面、平板、手机三个宽度。
- 检查浅色、深色、减少动效三种偏好。
- 检查最长中文标题、最长英文单词、空数据、错误状态、歌单不可用状态。
- 检查所有固定元素是否遮挡正文、目录、代码块和底部内容。
- 最终以截图对比验收首页、博客列表、文章详情、Wiki、Todo、AI 面板、音乐面板、小狗菜单。

## 7. 数据与内容管理

由于项目不使用后端，内容管理采用文件驱动：

- 新增博客：在本地 `content/blog/` 添加 Markdown。
- 新增 Wiki：在本地 `content/wiki/` 添加 Markdown。
- 新增 Todo：在本地 `content/todo/` 添加 Markdown 或 JSON。
- 新增主页白天背景：把图片放入 `public/backgrounds/home/day/` 并更新 `app/config/backgrounds.ts`。
- 新增主页黑夜背景：把图片放入 `public/backgrounds/home/night/` 并更新 `app/config/backgrounds.ts`。
- 新增通用背景：把图片放入 `public/backgrounds/shared/` 并更新前端配置。
- 新增音乐：优先更新 QQ 音乐歌单；本地备用音频放入 `public/music/` 并更新备用播放列表配置。
- 新增小狗资源：把图片、动画或序列帧放入 `public/pet/` 并更新宠物配置。
- 新增 AI 知识来源：优先从 `content/wiki/`、`content/blog/`、`content/todo/` 建立前端索引。
- 新增外部歌单：在 `app/config/music.ts` 维护 QQ 音乐公开歌单 ID、MetingJS 参数和备用歌单，避免把配置散落在组件模板里。

内容维护流程：

1. 在本地编辑器中新增或修改 `content/` 下的 Markdown 或 JSON。
2. 检查 frontmatter 是否完整，例如标题、日期、标签、状态、优先级。
3. 本地运行开发服务器预览页面。
4. 确认无误后提交到 Git。
5. 执行静态生成并部署 `.output/public`。

网页端内容边界：

- 网页端不写入 `content/`，也不能直接修改服务器文件。
- 网页端不提供文章编辑器、Wiki 编辑器、Todo 编辑器和富文本编辑器。
- 网页端的搜索、筛选、排序、展开、收起、主题切换、背景切换都属于浏览状态或偏好状态。
- `localStorage` 不作为正式内容来源，只作为偏好缓存。
- 如果未来确实需要在线写作，应单独引入后端、Git-based CMS、云函数或受保护的管理后台，并重新评估安全边界。

推荐在页面中表达内容维护方式：

- 关于页或页脚可以简短说明“内容由本地 Markdown 文件生成”。
- Wiki 和 Todo 的空状态可以提示“在本地 `content/wiki/` 或 `content/todo/` 添加文件后重新部署”。
- 不在页面主操作区放“去编辑”按钮，避免破坏只读产品心智。
- 面向自己维护时，可以在项目 README 和 `docs/` 中写详细步骤；面向访客时，页面只保留轻量说明。

Markdown Frontmatter 建议：

```yaml
---
title: 示例标题
description: 简短描述
date: 2026-05-15
tags:
  - nuxt
  - blog
category: frontend
draft: false
---
```

Todo Frontmatter 建议：

```yaml

status: in-progress
priority: high
startDate: 2026-05-15
targetDate: 2026-06-30
---
```

## 9. 腾讯云部署建议

推荐部署方式：

1. 本地执行构建：

```bash
npm run generate
```

2. 将 `.output/public` 上传到服务器目录，例如：

```text
/var/www/blog
```

3. 使用 Nginx 托管静态文件。

Nginx 配置要点：

- `root` 指向静态目录。
- `try_files` 支持前端路由刷新。
- 开启 gzip 或 brotli。
- 配置 HTTPS 证书。
- 页脚添加备案号。

示例方向：

```nginx
server {
    listen 80;
    server_name example.com www.example.com;

    root /var/www/blog;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

实际上线时需要替换为自己的域名，并补充 HTTPS 配置。

## 10. 风险与限制

纯前端限制：

- 无法进行真实用户登录。
- 无法在网页端直接安全写入服务器文件。
- 博客、Wiki、Todo 的新增和修改只能通过本地编辑文件后重新构建部署。
- 网页端不能可靠保存正式笔记、正式 Todo 和正式文章；不要设计会误导用户的保存按钮或可编辑任务控件。
- 联系表单不建议在无后端情况下直接实现提交。
- 音乐自动播放受浏览器策略限制。
- 真实 AI 模型调用不能在前端暴露私密 API Key。
- 模型名称、模型供应商配置、代理地址、鉴权令牌等敏感信息不能写入公开仓库。
- 小狗动画如果资源过大，会影响首屏加载和移动端性能。
- APlayer、MetingJS、外部 CDN、Meting API 或 QQ 音乐歌单服务可能不可用，必须准备降级方案。
- QQ 音乐歌曲可能因为版权、会员限制、地区限制或歌单私有状态无法播放，需要公开歌单和本地备用曲目兜底。

建议处理方式：

- 内容维护采用 Markdown 文件。
- 联系方式先做静态展示。
- 如果后续需要在线编辑博客、Todo 或 Wiki，再单独评估后端、CMS、Git-based CMS 或受保护的管理后台。
- AI 第一版先做本地知识库问答和交互壳，真实模型调用后续通过安全代理接入。
- 小狗资源优先使用 WebP、SVG、Lottie 或小尺寸序列帧，并设置懒加载。
- 音乐模块优先接入 APlayer + MetingJS + QQ 音乐公开歌单；外部服务不可用时降级到本地备用播放列表。
